# スキル: 記事レビュー

## 概要
記事ドラフトを複数観点からレビューし、具体的な修正指示を出力する。
サブエージェントに委任する形で使う。

## 呼び出し方

### サブエージェントへの指示テンプレート
```
~/agent/skills/article_review.md のレビュー手順に従い、以下の記事をレビューせよ。

対象記事: [記事ファイルパス]
ターゲット読者: [例: Java中級者、AWS初心者、エンジニア転職希望者]
記事種別: [無料ブログ / note有料記事 / Zenn技術記事]
シリーズ名: [あれば記載、なければ「単発」]

レビュー結果を ~/agent/content/reviews/[ファイル名]_review.md に保存せよ。
```

### 人気記事分析との連携
過去の人気記事分析結果がある場合は、以下も追加する:
```
人気記事の特徴分析: [分析ファイルパス]
```

## レビューチェックリスト

### 1. AIっぽさの排除

#### 定型文・決まり文句
- [ ] 「いかがでしたか」「いかがだったでしょうか」が使われていないか
- [ ] 「〜していきましょう」「〜してみましょう」が多用されていないか
- [ ] 「〜について解説します」が冒頭の定型パターンになっていないか
- [ ] 「まとめ」セクションが箇条書きの繰り返しになっていないか
- [ ] 「ぜひ〜してみてください」で締めていないか

#### 文末の単調さ
- [ ] 「〜です。」「〜ます。」が3文以上連続していないか
- [ ] 同じ文末表現が段落内で繰り返されていないか
- [ ] 体言止め・倒置・問いかけなど変化があるか

#### 過剰な丁寧さ・装飾
- [ ] 「非常に」「とても」「大変」などの強調語が多すぎないか
- [ ] 「〜することが可能です」→「〜できます」に簡潔化できないか
- [ ] 「〜という点において」等の冗長な接続が残っていないか
- [ ] 「重要なポイントとして」等、無意味な前置きがないか

#### 構成の不自然な整い方
- [ ] 全セクションが同じ文字数・同じ構造になっていないか
- [ ] 箇条書きの項目数が全セクションで揃いすぎていないか
- [ ] 「メリット3つ、デメリット3つ」のような機械的な対称性がないか

### 2. 技術的正確性

- [ ] コード例が実際に動作するか（構文エラー・非推奨APIがないか）
- [ ] バージョン番号・リリース日が最新情報と一致しているか
- [ ] 技術用語の定義が正確か（曖昧な比喩で誤解を招いていないか）
- [ ] 数値データ（性能比較・統計）の出典があるか、または妥当か
- [ ] 仕組みの説明が因果関係を正しく示しているか
- [ ] セキュリティ上の問題がある実装例を推奨していないか

### 3. 読みやすさ

#### 構造
- [ ] 1段落が5文以下に収まっているか
- [ ] 長い手順は箇条書きや番号付きリストになっているか
- [ ] 見出しだけ読んで記事全体の流れがわかるか
- [ ] コード例の前後に説明文があるか（コードだけ放り投げていないか）

#### 導入
- [ ] 最初の2文で「この記事を読むと何が得られるか」がわかるか
- [ ] 読者の課題・痛みから入っているか（自己紹介から始めていないか）

#### 情報の流れ
- [ ] 前提知識→本題→応用の順になっているか
- [ ] 専門用語の初出時に説明があるか
- [ ] セクション間の繋がりが自然か（唐突なトピック転換がないか）

### 4. 読者目線

- [ ] ターゲット読者の知識レベルに合っているか（簡単すぎ/難しすぎ）
- [ ] 読者が「で、どうすればいいの？」と思わない結論があるか
- [ ] 有料記事の場合、無料では得られない情報・価値があるか
- [ ] 読後に読者が具体的なアクションを取れるか
- [ ] タイトルが約束した内容を本文が果たしているか

### 5. シリーズ一貫性
（シリーズ記事の場合のみ）

- [ ] 文体（敬体/常体）が既存記事と統一されているか
- [ ] 用語の表記揺れがないか（例: サーバー/サーバ、JavaScript/JS）
- [ ] 前回記事の内容を不必要に繰り返していないか
- [ ] 前回記事への参照リンクがあるか

### 6. 人気記事の特徴との比較
（分析結果がある場合のみ）

- [ ] タイトルの形式が人気記事のパターンに沿っているか
- [ ] 記事の長さが人気記事の範囲内か
- [ ] 人気記事に共通する要素（具体例、コード量、図表）が含まれているか
- [ ] 人気記事で効果的だった導入パターンを参考にしているか

## AIっぽい表現リスト（検出用）

レビュー時にこのリストと照合し、該当箇所を指摘する。

### 危険度: 高（必ず修正）
| 表現 | 修正方向 |
|------|----------|
| いかがでしたか / いかがだったでしょうか | 削除、または具体的な問いかけに変更 |
| 〜していきましょう | 「〜する」「〜しよう」に簡潔化 |
| 〜することが可能です | 「〜できる」 |
| 網羅的に解説 / 徹底解説 | 具体的に何を説明するか書く |
| 〜について詳しく見ていきます | 削除、またはすぐ本題に入る |
| 〜のポイントを押さえましょう | 「〜が重要だ」等に言い換え |
| ぜひ参考にしてみてください | 具体的なアクションに変更 |
| 〜ではないでしょうか | 断定するか、具体的な根拠を示す |

### 危険度: 中（文脈次第で修正）
| 表現 | 判断基準 |
|------|----------|
| 非常に / とても / 大変 | 1記事3回以上なら減らす |
| 重要なポイントとして | 前置きせず直接書く方が良い場合は削除 |
| まず / 次に / 最後に | 連続使用3回以上なら変化をつける |
| 〜と言えるでしょう | 断定を避ける意図が明確なら許容 |
| 〜な方も多いのではないでしょうか | 根拠なく読者を想定している場合は削除 |

### 危険度: 低（パターン検出用）
| パターン | 注意点 |
|----------|--------|
| 全セクションが「〜とは」で始まる | 見出しの形式を変化させる |
| メリット/デメリットが同数 | 実態に合わせて非対称にする |
| 「3つの方法」「5つのコツ」系タイトル | 数字が内容に合っているか確認 |
| 冒頭で記事の構成を説明 | 短い記事なら不要、長い記事なら許容 |
